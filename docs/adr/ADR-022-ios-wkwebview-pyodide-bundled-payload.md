# ADR-022: iOS WKWebView + Pyodide Bundled Payload (Deterministic, App Store-Ready)

Date: 2026-02-19

## Context

We need an iOS wrapper app that runs a Pyodide (Python-in-WASM) payload inside `WKWebView`.

Constraints:

- App Store compliance: no runtime fetching of executable code that changes behavior (Guideline 2.5.2).
- Determinism: build artifacts (`dist/`) are not committed; instead they are generated by a pinned toolchain and lockfile, with a build manifest + hashes.
- Developer ergonomics: fast inner loop for _web payload_ iteration without rebuilding Swift, while keeping the production architecture the same (bundled assets only).

WKWebView-specific friction:

- `.wasm` loading often fails from `file://` due to missing/incorrect MIME type (`application/wasm`).

## Decision

1. The web payload is a Vite + TypeScript project under `libs/Fort-ios/`.
2. `dist/` is generated in CI (canonical) using a pinned Node toolchain (via `mise`) and locked dependencies (`package-lock.json`).
3. The build generates and embeds `dist/build-manifest.json` containing:
   - git SHA
   - lockfile hash
   - `dist/` content hash
   - tool versions
4. The iOS wrapper serves payload assets via `WKURLSchemeHandler` (custom scheme) rather than `file://`, to control MIME types (notably `.wasm` as `application/wasm`).
5. Navigation is strictly allowlisted via `WKNavigationDelegate` to prevent the app from acting as a general-purpose browser.
6. Debug-only Simulator iteration may support an explicit, opt-in container override directory for updated payload assets; Release/TestFlight must load from the app bundle only.

## Consequences

- Builds are reproducible and auditable, without committing build outputs.
- The iOS wrapper can reliably load WASM assets without running a local HTTP server.
- A DEBUG-only container override improves iteration speed while preserving the production architecture.

## Status

Accepted.

## Amendments

### 2026-02-21 — Wheel installation via `unpackArchive`; micropip abandoned

The web worker installs bundled `.whl` files (blake3, pychloride) using `pyodide.unpackArchive(buffer, 'wheel')` after fetching them via JS `fetch()` through the `app://` scheme handler. This is the only mechanism that works in the bundled-payload architecture.

**Why micropip was abandoned (two failed attempts):**

1. **micropip's internal `pyfetch()`** cannot reach `app://` scheme URLs — it uses a different fetch pathway than the JS `fetch()` API, so the scheme handler never receives the request.
2. **`emfs:` local path prefix** (staging wheels to Pyodide's virtual filesystem first) fails micropip's URL parser — it still attempts URL validation on the path.

**Current flow:**

1. `download-pyodide.sh` downloads Pyodide runtime + crypto wheels at build time (`make pyodide`).
2. `sync-payload.sh` copies `public/pyodide/` into `dist/pyodide/`, then syncs into `WebPayload/`.
3. At runtime, the worker calls `fetch('pyodide/wheels/blake3-....whl')` → `app://` scheme handler serves the file → `pyodide.unpackArchive(buffer, 'wheel')` extracts directly into site-packages.

**Dead code removed:** The micropip + packaging wheel downloads were removed from `download-pyodide.sh` since neither wheel is loaded by the worker.

### 2026-02-21 — `itms-services` auto-sanitization in `sync-payload.sh`

Bundled `python_stdlib.zip` contains the string `itms-services` in `urllib/parse.py`, which triggers automated App Store rejection. `sync-payload.sh` now automatically sanitizes this:

1. Unpacks `python_stdlib.zip` from `WebPayload/pyodide/`.
2. Replaces `itms-services` → `itms_services` in `lib/python3.13/urllib/parse.py` via `perl`.
3. Re-zips the archive in place.

No manual sanitization step is required before App Store submission.
