# ADR-024: Web Payload Build & Bundling Strategy

Date: 2026-02-19

## Context

The iOS wrapper at `libs/Fort-ios/` must embed the web payload built from `libs/Fort-ios/`. The payload is a Vite + TypeScript project that produces `dist/` output including HTML, JS, CSS, and (eventually) WASM assets.

Requirements:

- **Reproducible builds**: The same source + lockfile must produce the same output, locally and in CI.
- **Audit trail**: Every build must be traceable to a git commit, toolchain version, and content hash.
- **No committed build artifacts**: `dist/` must not be checked into the repo.
- **CI parity**: The exact same mechanism must work in local dev, Xcode Run Script phases, and GitHub Actions.
- **Clean sync**: Stale files in the wrapper's resource directory must be removed before copying fresh output.

ADR-022 established the principle of deterministic, bundled-only builds. This ADR specifies the concrete mechanism.

## Decision

1. **`sync-payload.sh`** is the canonical payload sync mechanism. It lives at `libs/Fort-ios/sync-payload.sh` and is the single source of truth for building and embedding the payload.

2. The script performs these steps in order:
   1. Build the web payload: `cd ../ios-pyodide-payload && npm ci && npm run build:ci`.
   2. Bundle Pyodide assets: copy `public/pyodide/` (runtime, stdlib, crypto wheels) into `dist/pyodide/` (~20MB).
   3. Verify `dist/build-manifest.json` exists and contains required fields.
   4. Sanitize `itms-services` in `python_stdlib.zip` (see ADR-022 amendment).
   5. Delete all contents of `libs/Fort-ios/WebPayload/` (clean stale files).
   6. Copy `dist/` contents into `libs/Fort-ios/WebPayload/`.
   7. Verify expected files exist (`index.html`, `build-manifest.json`).
   8. Print a summary (manifest SHA, file count, content hash).

3. **`WebPayload/`** is the directory inside the iOS wrapper project where payload assets are stored for Xcode to bundle. It is added to Xcode as a **folder reference** (blue folder) so nested directory structure is preserved.

4. **Xcode Run Script phase** may optionally invoke `sync-payload.sh`:
   - Input Files: `$(SRCROOT)/../ios-pyodide-payload/dist/build-manifest.json`
   - Output Files: `$(SRCROOT)/WebPayload/build-manifest.json`
   - Gated to Debug configuration or "when payload changed" to avoid unnecessary rebuilds.
   - The Xcode phase is a convenience, not the authority — the script is.

5. **`dist/build-manifest.json`** is generated by `libs/Fort-ios/tools/gen-build-manifest.mjs` and contains:
   - `schema`: Integer schema version (currently `1`).
   - `git_sha`: Current git commit hash.
   - `package_lock_sha256`: SHA-256 of `package-lock.json`.
   - `dist_tree_sha256`: SHA-256 of all files in `dist/` (excluding `build-manifest.json` itself).
   - `node_version`: Node.js version from `.tool-versions`.
   - `created_at`: ISO 8601 timestamp.

6. **Toolchain pinning**: `.tool-versions` (via `mise`) pins Node.js version. `package-lock.json` pins all npm dependencies. `npm ci` (not `npm install`) is the only allowed install command.

## Alternatives Rejected

| Alternative | Why Rejected |
|---|---|
| **Symlink `dist/` into Xcode project** | Fragile across Xcode build system; Xcode may not detect file changes correctly; breaks folder reference semantics |
| **Commit `dist/` to the repo** | Non-deterministic (depends on who built it); bloats repo history; violates ADR-022 |
| **Xcode Run Script as sole mechanism** | CI divergence — CI cannot use the same Xcode phase; no standalone script for local testing |
| **Copy `dist/` manually** | Not reproducible; misses manifest verification; leaves stale files |
| **npm `postbuild` script that copies** | Couples payload build to wrapper structure; breaks payload independence |

## Consequences

- Single script (`sync-payload.sh`) is the identical mechanism used by developers locally, Xcode build phases, and CI pipelines.
- `build-manifest.json` provides a verifiable audit trail for every build (traceable to exact commit and dependencies).
- Payload project remains independently buildable and testable — it doesn't know about the iOS wrapper.
- Wrapper project can be built without rebuilding the payload (if `WebPayload/` already contains current assets).
- Trade-off: Developers must remember to run `sync-payload.sh` after payload changes (or rely on the optional Xcode Run Script phase).

## Status

Accepted.

## Amendments

### 2026-02-20 — Node version check loosened for local dev

The `sync-payload.sh` Node version check was changed from an exact string match to a major.minor comparison for local development. CI retains the exact match requirement (enforced via `mise`).

- Local dev: warns if major.minor differs (patch upgrades like `22.12.0` → `22.13.0` are safe)
- CI (`CI=true` or `GITHUB_ACTIONS=true`): hard error on any version mismatch

### 2026-02-20 — `Makefile` added as entry point

`libs/Fort-ios/Makefile` added. `make sync` is now the preferred invocation of `sync-payload.sh`. The script itself is unchanged; the Makefile provides discoverability via `make help`.

### 2026-02-21 — Pyodide asset bundling and `itms-services` sanitization

Two additional steps added to `sync-payload.sh`:

1. **Pyodide bundling**: After building the web payload, the script copies `public/pyodide/` (WASM runtime, Python stdlib, crypto wheels) into `dist/pyodide/`. These assets are downloaded once via `make pyodide` (runs `download-pyodide.sh`).
2. **`itms-services` sanitization**: Automatically unpacks `python_stdlib.zip`, replaces `itms-services` → `itms_services` in `urllib/parse.py`, and re-zips. This prevents automated App Store rejection.

The step count in Decision item 2 was updated from 6 to 8 to reflect these additions.

### 2026-02-21 — `build-manifest.json` field names corrected

Decision item 5 was updated to match the actual field names used by `gen-build-manifest.mjs`. All fields use `snake_case` and a `schema` version field was added.
