# ADR-028: iOS SwiftLint & Type-Inference Strategy

Date: 2026-02-21

## Context

The iOS KeriWallet project has no automated linting for Swift source. SourceKit-LSP provides inlay hints (virtual inline annotations showing inferred types and parameter names), but without a linter there is:

- No enforcement of the project's no-force-unwrap (`!`) rule from `ios-swift-coding.instructions.md`.
- No guardrail against `// TODO` / `// FIXME` comments silently replacing the required `#warning("TODO: ...")` compiler directive pattern.
- No consistent style enforcement across contributors.
- Risk of over-annotating types (e.g., `let x: Int = 5`) which fights Swift's idiomatic type inference and adds noise.

Swift's official position — documented in the [API Design Guidelines](https://www.swift.org/documentation/api-design-guidelines/) ("prefer implicitness over explicitness unless being explicit improves readability") and the [Google Swift Style Guide](https://google.github.io/swift/) ("avoid redundancy, avoid ambiguity, and prefer implicitness over explicitness") — is that type inference for locals and obvious initializers is idiomatic.

The question is: which linting approach and type-annotation strategy best serves this project?

## Decision

### 1. Adopt SwiftLint (v0.63.2+)

Install via Homebrew (`brew install swiftlint`). Configuration lives in `libs/Fort-ios/.swiftlint.yml`.

**Run locally:** `make lint` from `libs/Fort-ios/`.
**VS Code integration:** `vknabel.vscode-swiftlint` extension runs on save.
**CI:** Add `swiftlint lint --config .swiftlint.yml --strict` to CI pipeline.

### 2. Type-Inference Strategy: Idiomatic Inference + Redundancy Guard

Enable the `redundant_type_annotation` opt-in rule with `consider_default_literal_types_redundant: true`. This:

- **Warns** when a type annotation repeats what the compiler already infers (e.g., `let name: String = "hello"` → remove `: String`).
- **Flags** default literal types too (e.g., `let count: Int = 0` → remove `: Int`, since `Int` is the default integer literal type).
- **Does NOT require** explicit types everywhere — it only catches redundant annotations.

This aligns with Swift API Design Guidelines and prevents the codebase from drifting toward C-style verbosity.

### 3. Security-Aligned Opt-in Rules

| Rule | Default? | Why enabled |
|------|----------|-------------|
| `force_unwrapping` | Opt-in | Enforces the zero-tolerance `!` ban from `ios-swift-coding.instructions.md` |
| `redundant_type_annotation` | Opt-in | Prevents over-annotation; keeps inference idiomatic |
| `force_cast` | Default | Already enabled — catches `as!` force casts |
| `force_try` | Default | Already enabled — catches `try!` force attempts |
| `todo` | Default | Catches `// TODO:` / `// FIXME:` comments — aligns with `#warning("TODO: ...")` directive requirement |

### 4. Auto-Generated File Exclusion

`BridgeContract.swift` (generated by `tools/gen-bridge-contract.mjs` from `bridge-contract.json`) is excluded from linting. Codegen output would trigger false positives for style rules that don't apply to machine-generated code.

### 5. Inlay Hints Configuration

VS Code inlay hints (SourceKit-LSP) are set to `offUnlessPressed` for Swift files. This means:

- **Default view:** Clean code without type annotations cluttering the editor.
- **On demand:** Hold Ctrl+Alt to reveal inferred types/parameter names for orientation.

This balances discoverability with visual cleanliness.

## Alternatives Rejected

### A. `explicit_type_interface` rule (require types everywhere)

SwiftLint offers an `explicit_type_interface` opt-in rule that _requires_ explicit type annotations on all declarations. Rejected because:

- Fights Swift's core design philosophy ("prefer implicitness over explicitness").
- Would force `let x: Int = 5` everywhere, which `redundant_type_annotation` correctly flags as noise.
- Google Swift Style Guide explicitly marks such patterns with ⛔️.
- Would generate hundreds of violations against the existing (correctly idiomatic) codebase.

### B. No linter (status quo)

Rejected because the project's `ios-swift-coding.instructions.md` already mandates rules (no `!`, no `// TODO`, no `print()`) that are only enforced by human review. SwiftLint automates enforcement and catches violations before code review.

### C. SwiftFormat instead of SwiftLint

SwiftFormat focuses on formatting (whitespace, braces) rather than lint rules (force unwraps, TODOs). SwiftLint covers both formatting and semantic linting. SwiftFormat could be added later for auto-formatting, but SwiftLint is the minimum viable tool for enforcing the project's existing coding standards.

## Consequences

### Positive

- Force unwraps, force casts, and force-try violations are caught automatically — no longer rely solely on code review.
- `// TODO` / `// FIXME` comments are flagged, reinforcing the `#warning("TODO: ...")` pattern.
- Type inference stays idiomatic — `redundant_type_annotation` prevents the codebase from becoming over-annotated.
- `make lint` provides a one-command quality gate.
- CI can fail on lint violations (`--strict` mode).

### Negative

- Developers must install SwiftLint (`brew install swiftlint`) — documented in Makefile `help` and onboarding.
- Auto-generated files must be excluded in `.swiftlint.yml` — requires maintenance when new codegen is added.

### Neutral

- Inlay hints remain available on-demand (Ctrl+Alt) for type discovery without polluting the visual editor.
- Existing codebase must pass lint before this is merged — initial run required to establish a clean baseline.
